# 1\. 프로젝트 개요

## 1.1 서비스명

CoinBot

## 1.2 한 줄 정의

한국 5개 거래소(빗썸·업비트·코인원·코빗·고팍스)의 에어드랍·N빵 이벤트를 자동 수집하고, 사용자 거래소 계정을 통해 이벤트 참여(자동 매수)를 돕는 이벤트 자동 참여 봇.

## 1.3 서비스 목적

* 5개 거래소 이벤트 공지를 12시간마다 자동 수집한다.
* 수집된 이벤트를 자동 승인하거나 관리자가 검토하여 이벤트 게시판에 등록한다.
* 이벤트 등록 시 API를 등록한 전체 사용자를 대상으로 자동 매수를 실행한다.
* 거래 결과를 텔레그램 알림으로 즉시 전달한다.
* 모바일 웹에서도 쉽게 사용할 수 있도록 단순한 UI로 제공한다.

## 1.4 서비스 성격

* 공개 SaaS가 아닌 소규모 Private Tool
* 사용자 범위: 재한님 + 초대된 사용자 (다음 단계: 사용자 초대/회원 설정)
* URL로 접속하는 웹 서비스 (HTTPS 적용 완료)
* 배포: AWS EC2 (`43.203.100.239`, Amazon Linux 2, pm2, 포트 3000)
* DB: Supabase
* 스케줄러: pm2 cron (12시간마다 이벤트 크롤링)
* 거래 방식: 시장가만 지원

# 2\. 페르소나

## 2.1 Primary Persona: 관리자 / 서비스 오너

이름: 김재한

### 특징

* 여러 코인 거래소 계정을 관리한다.
* 자동화 서비스와 투자 서비스에 관심이 높다.
* 개발 지식은 많지 않지만 서비스 기획과 운영 의사결정은 직접 한다.
* 실제 운영 시 친구들의 API 정보를 받아 직접 관리할 예정이다.

### 문제

* 여러 계정을 각각 접속해 수동 거래하는 것이 번거롭다.
* 특정 시간에 거래를 놓치지 않고 실행하고 싶다.
* 반복 업무를 자동화하고 싶다.

### 목표

* 한 번의 입력으로 여러 계정에 동일한 거래를 실행한다.
* 스케줄로 자동 거래를 수행한다.
* 최소한의 클릭으로 빠르게 거래를 설정한다.

## 2.2 Secondary Persona: 친구 사용자

이름: 친구 A / 친구 B / 친구 C

### 특징

* 재한님이 만든 서비스를 사용한다.
* 복잡한 설정은 선호하지 않는다.
* 최대한 간단한 입력과 직관적인 UI를 원한다.

### 목표

* 로그인 후 필요한 거래만 쉽게 등록한다.
* 복잡한 API 관리 없이 거래 기능만 사용한다.

# 3\. 제품 원칙

## 3.1 MVP 원칙

이 서비스는 처음부터 많은 기능을 넣지 않고, 꼭 필요한 기능만 구현한다.

## 3.2 단순성 원칙

* 화면은 Single Page 중심으로 단순하게 구성한다.
* 사용자가 직접 API를 관리하지 않도록 한다.
* 시장가 거래만 지원한다.
* 결과는 화면에서 즉시 확인하고, 로그 저장은 2단계로 미룬다.

## 3.3 보안 원칙

* API Key는 본인 또는 관리자만 관리한다. (일반 사용자는 본인 것만, 관리자는 전체)
* DB 저장 시 AES-256-GCM으로 암호화한다.
* API 발급 시 입출금 권한은 제외한다.
* 조회/거래 권한만 있는 API를 사용한다.

# 4\. 서비스 범위

## 4.1 구현 완료 범위

* ID/비밀번호 로그인 (세션 기반)
* 사용자 본인 거래소 API 계정 등록/조회/삭제 UI
* 관리자 전용 페이지(/admin): 모든 사용자 계정 대리 등록/조회/삭제
* 수동 즉시 실행 (지금 실행, 실행 전 검증 포함)
* 스케줄 등록/실행 (trade_jobs, pm2 cron)
* **이벤트 자동 수집**: 5개 거래소 공지 크롤링 (12시간 간격, pm2 cron)
* **텔레그램 알림**: 신규 이벤트 수집 알림, 거래 결과 알림
* **관리자 이벤트 검토 UI**: 수집된 이벤트 승인/거절, 이벤트 게시판 등록
* **크롤러 키워드 관리**: 포함/제외 키워드 UI에서 관리 (DB 저장)
* **수집 이력 조회**: crawl_logs 기반 최근 20건 조회
* GitHub Actions CI/CD: main push → EC2 자동 배포
* React ErrorBoundary: 흰 화면 방지
* 이미지 최적화 (WebP 압축)

## 4.2 미구현 범위 (제외 또는 보류)

* 지정가 거래
* 분할 매수/매도
* 손절 기능
* API 정보 수정 화면 (등록/삭제만 지원)
* 실행 결과 다운로드
* 업비트 이벤트 수집 (EC2 IP Cloudflare 차단 — Lambda@Edge로 해결 예정)

## 4.3 다음 단계 개발

* **완전 자동화**: 이벤트 자동 승인 (키워드 조건 강화) + 이벤트 등록 시 전체 사용자 자동 매수
* **업비트 수집 해결**: Lambda@Edge / CloudFront 프록시 경유
* **사용자 초대/회원 설정**: 초대 링크, 회원 관리 UI

# 5\. 주요 사용자 시나리오

## 5.1 첫 사용 시나리오

1. 사용자가 서비스 URL로 접속한다.
2. 로그인 또는 회원가입을 진행한다.
3. 로그인 후 메인 화면에 진입한다.
4. 관리자가 사전에 등록한 API 계정 목록이 거래소별로 노출된다.
5. 사용자는 거래소, 코인, 금액, 계정, 실행 방식을 선택한다.
6. 거래를 저장하거나 즉시 실행한다.

## 5.2 지금 실행 시나리오

1. 사용자가 거래 입력을 완료한다.
2. \[지금 실행] 선택
3. 실행 전 검증 화면으로 이동
4. 거래소, 계정, 주문 내용, 잔고, 가능 여부 확인
5. 사용자가 \[실행] 클릭
6. 서버가 각 계정에 대해 거래 실행
7. 결과 화면에서 계정별 성공/실패 확인
8. 자산 현황을 함께 확인

## 5.3 스케줄 실행 시나리오

1. 사용자가 거래 입력을 완료한다.
2. From / To / 시간 입력 후 \[스케줄 등록]
3. trade\_jobs에 저장
4. 외부 cron 서비스가 정해진 시각에 실행 API 호출
5. 서버가 해당 시점에 실행 대상인지 판별
6. 중복 실행이 아니면 거래 실행
7. MVP에서는 결과를 화면 저장하지 않고 서버 실행만 수행

## 5.4 거래 수정 시나리오

1. 사용자가 거래 목록을 연다.
2. 특정 행을 선택한다.
3. 수정 또는 삭제 버튼을 누른다.
4. 수정 시 날짜/시간만 변경 가능하다.
5. 저장 후 목록에 반영된다.

# 7\. 상세 기능 정의

## 7.1 로그인 / 회원가입

* ID와 비밀번호 기반 인증
* 비밀번호는 hash 저장
* 로그인 성공 시 세션 발급
* 세션은 브라우저 종료 시 만료

## 7.2 API 계정 관리

운영 정책

* 일반 사용자: 로그인 후 "거래소 등록" 탭에서 본인의 거래소 API 계정을 직접 등록/조회/삭제한다.
* 관리자: `/admin` 페이지에서 모든 사용자의 API 계정을 대리 등록/조회/삭제할 수 있다.
* 관리자 지정은 `.env.local`의 `ADMIN_USER_ID` 환경변수로 수행한다.
* API 수정 화면은 만들지 않는다. 수정이 필요하면 삭제 후 재등록한다.
* 등록 시 API 키 유효성을 잔고 조회로 검증한다.

보안 정책

* API는 발급 시 입출금 권한 제외
* 조회/거래 권한만 허용
* access\_key / secret\_key는 AES-256-GCM 암호화 저장 (필수)
* API 원문은 어느 UI에도 노출하지 않는다 (password input + 저장 후 조회 불가)

## 7.3 거래 입력

* 거래소 1개 선택
* 코인 입력
* 거래 방식 선택
* 금액 입력
* 계정 다중 선택
* 즉시 실행 또는 스케줄 등록

# 8\. 상세 로직

## 8.1 코인 처리 로직

1. 사용자가 BTC, ETH, USDT 등의 코인명/코드 입력
2. 서버가 선택 거래소의 마켓 목록 조회
3. 입력값을 거래소에서 사용하는 KRW 마켓 코드로 매핑
4. 유효하지 않으면 에러 반환

## 8.2 금액 → 수량 변환 로직

1. 사용자가 KRW 금액 입력
2. 서버가 현재가 조회
3. 거래 수량 계산
4. 시장가 주문 요청 생성

## 8.3 즉시 실행 로직

1. 거래 요청 조회
2. 선택된 account\_ids 기준으로 API 계정 목록 로드
3. 각 계정에 대해:

   * 잔고 조회
   * 최소 금액 확인
   * 코인 유효성 확인
   * 시장가 주문 실행
4. 결과 집계
5. 화면 반환

의사 코드:

```text
for account in selected\\\_accounts:
    balance = get\\\_balance(account)
    if balance insufficient:
        result = FAIL("잔고 부족")
    elif amount < 5100:
        result = FAIL("최소 금액 미달")
    else:
        market = resolve\\\_market(exchange, coin)
        qty = calc\\\_quantity(amount, current\\\_price)
        place\\\_market\\\_order(account, market, qty, side)
        result = SUCCESS or FAIL(reason)
```

## 8.4 동시 실행 로직

정책

* 사용자 결정사항 기준: 전부 동시에 실행
* 구현 기준: Promise.all 방식

주의

* 제3자 관점에서 rate limit 리스크가 있으므로 내부 제한 로직을 추후 고려할 수 있다.
* 하지만 최종 결정사항은 현재 "전부 동시에 실행"이다.

## 8.5 스케줄 실행 로직

외부 cron 서비스가 실행 API를 호출한다.

실행 조건

1. 오늘 날짜가 schedule\_from \~ schedule\_to 범위 내에 있어야 한다.
2. 현재 시간이 schedule\_time과 일치해야 한다.
3. last\_executed\_at이 오늘이 아니어야 한다. (중복 실행 방지)

의사 코드:

```text
if today in \\\[schedule\\\_from, schedule\\\_to]:
    if now\\\_time == schedule\\\_time:
        if last\\\_executed\\\_at != today:
            run\\\_trade\\\_job()
            update last\\\_executed\\\_at
```

## 8.6 스케줄 실패 처리

최종 결정사항

* 실패 시 1회 재시도

# 9.4 거래소별 API 연동 방식 (2026-04-05 추가)

| 거래소 | 인증 방식 | 구현 위치 | 마켓 심볼 형식 |
|------|------------|------|------|
| 빗썸 | V2 JWT Bearer (HS256) | `src/lib/bithumb-v2.ts` 직접 구현 | `KRW-BTC` |
| 업비트 | ccxt | `src/lib/exchange.ts` | `BTC/KRW` |
| 코인원 | ccxt | `src/lib/exchange.ts` | `BTC/KRW` |
| 코빗 | ccxt | `src/lib/exchange.ts` | `BTC/KRW` |
| 고팍스 | ccxt | `src/lib/exchange.ts` | `BTC/KRW` |

* 빗썸 V2는 ccxt 4.x가 미지원 → `bithumb-v2.ts`에 직접 구현
* JWT 페이로드: `access_key`, `nonce`, `timestamp`, (POST 시) `query_hash`, `query_hash_alg`
* query\_hash: 요청 파라미터 query string의 SHA512 hex 값
* 최소 주문 금액: 5,100원 (코인원 이벤트성 거래 기준 포함)

# 10\. 시스템 아키텍처

## 10.1 구성

* Frontend: Next.js App Router 기반 웹앱
* Hosting: AWS EC2 (`43.203.100.239`, Amazon Linux 2, Node.js 20, pm2)
* Backend: Next.js API Routes (포트 3000)
* DB: Supabase
* Scheduler: pm2 cron (크롤링 12시간, trade_jobs 스케줄 실행)
* Exchange API 연동: 빗썸 V2 JWT 직접 구현 + 나머지 4개 ccxt
* CI/CD: GitHub Actions → EC2 SSH 자동 배포
* HTTPS: DuckDNS + Let's Encrypt (적용 완료)

## 10.2 아키텍처 흐름

```text
사용자 브라우저 (HTTPS)
↓
EC2 (Next.js, pm2, 포트 3000)
↓
API Routes
↓
Supabase
↓
코인 거래소 API
```

이벤트 크롤링 흐름

```text
pm2 cron (0 0,12 * * *)
↓
POST /api/cron/crawl-events (CRON_SECRET 인증)
↓
5개 거래소 Promise.allSettled 동시 크롤링
↓
crawled_events 저장 (pending)
↓
관리자 검토 → 승인 시 announcements 등록
↓
텔레그램 알림 발송
```

스케줄 거래 흐름

```text
pm2 cron
↓
실행 API
↓
trade_jobs 조회
↓
조건 확인
↓
거래소 API 실행
```

완전 자동화 목표 흐름 (다음 단계)

```text
이벤트 수집
↓
키워드 자동 승인 (엄격한 조건)
↓
announcements 등록
↓
모든 사용자 API 계정 대상 자동 매수 (Promise.all)
↓
실패 시 재시도 → 텔레그램 알림
```

# 13\. 운영 관점 결정사항 (Decision Log)

1. 서비스 형태는 웹 서비스 + URL 배포로 확정
2. 로그인 방식은 ID/비밀번호
3. 사용자 ID는 직접 입력
4. 비밀번호는 최초 입력 후 변경 가능, 2회 입력으로 오입력 방지
5. 세션 유지 시간은 브라우저 종료 시까지
6. 시장가 거래만 지원
7. 거래소는 라디오 버튼으로 1개 선택
8. 코인은 직접 입력, 예시 문구 표시
9. 금액은 KRW 입력, 기본값 5,100원
10. 5,100원 미만일 때만 최소 금액 오류 처리
11. API 정보는 재한님이 DB에 직접 입력
12. 사용자 화면에서 API 관리 기능은 제공하지 않음
13. 계정 이름은 관리자가 DB에 직접 입력한 값 사용
14. 사용자는 선택된 거래소에 대해 등록된 계정 목록을 체크박스로 선택
15. 거래 목록은 여러 건 저장 가능
16. 거래 목록에서 라인 선택 후 수정/삭제 가능
17. 수정은 날짜와 시간만 가능
18. 지금 실행과 스케줄 등록 둘 다 필요
19. 실행 전 검증 화면은 초기 요구사항의 중간 화면으로 확정
20. 결과 화면은 즉시 실행 시 화면에서 바로 보여줌
21. 스케줄 결과 알림은 2단계 기능
22. 로그 저장은 2단계 기능
23. 스케줄은 외부 cron 사용
24. 스케줄 실패 시 1회 재시도
25. 동시 실행은 전부 동시에 실행으로 확정
26. API 키 권한은 입출금 제외
27. 결과는 MVP에서 DB 저장하지 않음
28. API 수정은 별도 UI 없이 DB에서 직접 관리
29. exchange\_accounts와 trade\_accounts는 합치지 않고, trade\_accounts는 만들지 않는 단순 구조 채택
30. 사용자 API 관리 UI는 MVP에 포함 (일반 사용자는 본인 것만, 관리자는 /admin에서 전체 관리)
31. 관리자는 `.env.local`의 `ADMIN_USER_ID` 환경변수로 지정
32. 거래소 표시 순서는 빗썸 → 업비트 → 코인원 → 코빗 → 고팍스로 고정
33. 빗썸은 V2 API(JWT Bearer) 직접 구현 사용, 나머지 4개는 ccxt 라이브러리 사용
34. Supabase는 `claude_dev` 커스텀 스키마 사용 (`NEXT_PUBLIC_DB_SCHEMA` 환경변수)
35. 배포는 Vercel이 아닌 AWS EC2 + pm2 (포트 3000)로 확정 전환
36. 이벤트 크롤링은 pm2 cron으로 12시간 간격 실행 (`0 0,12 * * *`)
37. 텔레그램 알림 MVP에 포함 (신규 이벤트 수집·거래 결과)
38. HTTPS는 DuckDNS + Let's Encrypt로 적용 완료
39. 업비트 수집 차단(EC2 IP → Cloudflare WAF) 해결책: Lambda@Edge / CloudFront
40. 자동 승인 방향: 관리자 검토 제거 → 키워드 조건을 더 엄격하게 하여 자동 승인
41. 완전 자동화 매수: 이벤트 등록 시 모든 사용자 자동 매수 (사용자별 ON/OFF 없음)
42. 자동 매수 실패 처리: N회 재시도 후 텔레그램 알림
43. 다음 개발 우선순위: 완전 자동화 → 업비트 수집 해결 → 사용자 초대/회원 설정

# 16\. 성공 기준

이 문서는 아래 조건을 만족하면 성공한 MVP로 본다.

1. 사용자가 ID/비밀번호로 로그인할 수 있다.
2. 거래소를 1개 선택하고 코인/금액/계정을 입력할 수 있다.
3. 저장된 거래 요청을 목록으로 볼 수 있다.
4. 거래 요청을 수정(날짜/시간) 또는 삭제할 수 있다.
5. 즉시 실행 전에 검증 화면에서 잔고/가능 여부를 볼 수 있다.
6. 즉시 실행 시 계정별 결과를 확인할 수 있다.
7. 스케줄 등록이 가능하고, 기간/시간 조건에서 실행된다.
8. 동일 날짜 중복 실행이 방지된다.
9. 최소 금액/코인 오류/API 오류가 명확한 문구로 표시된다.
10. 일반 사용자는 본인의 거래소 계정을 등록/삭제할 수 있다.
11. 관리자(/admin)는 모든 사용자의 거래소 계정을 대리 등록/삭제할 수 있다.
12. API 원문은 어느 UI에도 노출되지 않는다.

# 18\. 이벤트 크롤링 시스템 (2026-04-14 추가)

## 18.1 개요

5개 거래소 공지사항에서 에어드랍/N빵 이벤트를 자동 수집하여 관리자 검토 또는 자동 승인 후 게시판에 등록하는 반자동/완전자동 워크플로우.

## 18.2 DB 테이블

### crawled_events

```sql
CREATE TABLE crawled_events (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  exchange TEXT NOT NULL,
  source_id TEXT NOT NULL,
  title TEXT NOT NULL,
  url TEXT,
  crawled_at TIMESTAMPTZ DEFAULT now(),
  status TEXT DEFAULT 'pending' CHECK (status IN ('pending', 'approved', 'rejected')),
  reviewed_by TEXT,
  reviewed_at TIMESTAMPTZ,
  published_event_id UUID REFERENCES announcements(id) ON DELETE SET NULL,
  UNIQUE (exchange, source_id)
);
```

### crawl_logs

```sql
CREATE TABLE crawl_logs (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  triggered_by TEXT NOT NULL CHECK (triggered_by IN ('cron', 'manual')),
  started_at TIMESTAMPTZ DEFAULT now(),
  total_collected INTEGER DEFAULT 0,
  new_count INTEGER DEFAULT 0,
  telegram_sent BOOLEAN DEFAULT false,
  errors JSONB DEFAULT '[]'
);
```

### crawler_keywords

포함/제외 키워드를 DB에 저장. 관리자 UI에서 실시간 편집 가능 (재배포 불필요).

## 18.3 거래소별 수집 방식

| 거래소 | 방식 | 상태 |
|--------|------|------|
| 빗썸 | 공식 REST API | ✅ 정상 |
| 업비트 | 비공식 API | ❌ EC2 IP Cloudflare 차단 |
| 코인원 | HTML 파싱 (`__NEXT_DATA__`) | ✅ 정상 |
| 코빗 | HTML 파싱 (`__NEXT_DATA__`) | ✅ 정상 |
| 고팍스 | 공식 REST API | ✅ 정상 |

## 18.4 재시도 로직

* `withRetry(fn, { retries: 2, delayMs: 1000, label })` — 지수 백오프 (1s→2s→4s)
* 빗썸·코인원·코빗·고팍스 적용 (업비트는 차단으로 제외)
* 5분 중복 실행 방지: `crawl_logs` 최근 5분 내 실행 여부 확인

## 18.5 자동 승인 방향 (다음 단계)

현재: 관리자가 수동 검토 → 승인/거절

목표: 키워드 조건을 충분히 엄격하게 설정 → 수집 즉시 자동 승인 + 이벤트 게시판 등록

리스크 대응:
* 포함 키워드를 "에어드랍", "N빵", "무료 지급" 등 명확한 이벤트 용어로 한정
* 제외 키워드로 점검·공지·거래량 관련 게시글 필터링
* 자동 승인된 이벤트는 관리자 UI에서 언제든 거절(삭제) 가능

# 19\. 완전 자동화 매수 (다음 단계)

## 19.1 개념

이벤트가 announcements에 등록되는 순간, API를 등록한 모든 사용자를 대상으로 자동으로 매수 실행.

## 19.2 실행 흐름

```text
이벤트 승인/등록 (자동 또는 수동)
↓
전체 사용자 exchange_accounts 조회 (해당 거래소)
↓
Promise.all 동시 매수 실행
↓
성공: DB 기록 + 텔레그램 알림
실패: N회 재시도 → 여전히 실패 시 텔레그램 알림 (계정별 실패 원인 포함)
```

## 19.3 실패 처리 정책

* 재시도: 즉시 1~2회 재시도 (지수 백오프)
* 재시도 후 실패: 텔레그램으로 실패 계정 + 원인 알림
* 부분 성공: 성공한 계정과 실패한 계정을 분리하여 알림

## 19.4 사용자 선택권

* 사용자별 ON/OFF 없음 → API를 등록한 모든 계정이 자동 매수 대상
* 특정 거래소를 빠지려면 해당 거래소 API 계정을 삭제하면 됨

# 20\. 업비트 수집 해결 방안 (미구현)

## 현황

EC2 IP(`43.203.100.239`)가 Cloudflare WAF에 차단되어 업비트 비공식 API 접근 불가.

## 목표 해결책: Lambda@Edge / CloudFront

* AWS Lambda@Edge 또는 CloudFront를 경유하여 국내 IP처럼 보이게 처리
* 또는 국내 ISP 경유 프록시 서버 사용
* 구현 시 업비트 크롤러(`src/lib/crawlers/upbit.ts`) 프록시 URL 설정만으로 활성화 가능
